# Encode Lab 业务逻辑与功能模块调整

- 文档版本：`v0.1`
- 更新时间：`2026-05-08`
- 文档类型：`产品模块梳理`
- 适用阶段：`V1 功能收敛与后续模块重排`
- 关联规格：[`ffmpeg-v1-功能规格.md`](./ffmpeg-v1-功能规格.md)

## 1. 调整目标

Encode Lab 的核心业务不是“配置 FFmpeg 参数”，而是帮助用户完成一条稳定的转码决策链路：

1. 识别素材和机器能力。
2. 生成或复用一套参数方案。
3. 通过预览验证效果、体积和速度预期。
4. 将确认后的方案应用到单文件或批量文件。
5. 在任务执行中持续观察、控制和复盘结果。

因此，功能模块应围绕“用户如何完成转码决策与执行”组织，而不是围绕前端页面或后端命令拆分。

## 2. 当前问题

### 2.1 页面结构按实现阶段拆分

当前前端导航为：

1. `任务配置`
2. `预览`
3. `任务中心`
4. `模板`
5. `设置`

这套结构能覆盖 V1 功能，但用户心智容易被拆散：

1. `任务配置` 和 `预览` 实际属于同一个“调参工作台”。
2. `模板` 不是独立业务终点，而是“参数方案资产”的管理入口。
3. `任务中心` 既承担运行控制，又承担历史复盘，但当前命名只强调任务列表。
4. `设置` 中的 FFmpeg 探测、并发数和默认目录实际影响整个转码能力边界。

### 2.2 业务对象边界不够清晰

现有代码中已经存在这些业务对象：

1. 素材：`sourceFilePath`、`VideoMetadataResult`、`read_video_metadata`
2. 参数草稿：`TaskDraftContext`、`TaskDraftSnapshot`
3. 参数方案：`Template`
4. 转码任务：`TaskConfig`
5. 执行记录：`JobHistory`
6. 运行指标：`JobMetricsEvent`
7. 本机能力：`FfmpegProbeResult`、`EncoderCapabilityResult`
8. 未来节点能力：`NodeDescriptor`、`FileLocation`、`TransferPlan`

问题在于这些对象目前分散在页面、上下文、后端命令和 V2 类型中，产品层没有统一说明“谁是用户资产、谁是运行过程、谁是系统能力”。

### 2.3 V2 分布式概念已经提前渗入类型层

`FileLocation`、`NodeDescriptor`、`TransferPlan` 已经在前端类型和后端模型中出现。它们对未来扩展有价值，但 V1 当前仍是本机路径、本机 FFmpeg 和本地 JSON 队列。

后续模块调整应避免让用户在 V1 中感知“节点/分布式”的复杂度，只在内部模型上保留兼容字段。

## 3. 新的业务主线

V1 应聚焦一条主线：

```text
素材导入 -> 参数方案 -> 效果预览 -> 入队执行 -> 过程控制 -> 结果复盘
```

这条主线对应的用户问题是：

1. 这个视频是什么规格？
2. 这台机器能用哪些编码器？
3. 我应该用哪套参数？
4. 这套参数的画质、体积和速度是否可接受？
5. 我能不能把它批量应用到更多文件？
6. 转码过程是否正常，失败时能否定位原因？

所有功能模块都应服务这六个问题。

## 4. 调整后的功能模块

### 4.1 工作台

工作台承载从素材到入队前的完整调参流程，是 V1 的主入口。

职责：

1. 选择源视频或拖入视频。
2. 读取元数据后调节视频、音频、容器和高级参数。
3. 参数配置页只负责调整参数，并提供返回源文件选择页的入口。
4. 预览校验页生成预览帧，并在同页确认源文件、当前参数和输出决策后加入队列。
5. 将当前参数保存为方案，供后续复用。

当前代码映射：

1. `src/context/TaskDraftContext.tsx`
2. `src/pages/SourceSelectPage.tsx`
3. `src/pages/TaskConfigPage.tsx`
4. `src/pages/PreviewPage.tsx`
5. `src/components/workbench/SourceVideoCard.tsx`
6. `src/components/workbench/ComparePreviewPlayer.tsx`
7. `src-tauri/src/commands/preview.rs`
8. `src-tauri/src/commands/transcode.rs`

调整建议：

1. 产品命名从“任务配置 + 预览”收敛为“工作台”。
2. 前端可以继续保留内部路由，但侧边导航不应把调参和预览拆成两个同级业务模块。
3. 工作台内用步骤表达状态：`素材`、`参数`、`预览`、`确认任务`。

当前落地状态：

1. 工作台主导航默认进入 `/source`，第一步页面只负责源文件选择、拖拽和素材读取。
2. `/task-config` 是第二步参数配置页，只展示源素材摘要、时长截取和参数表单，并可返回 `/source` 重新选择源文件。
3. `/preview` 是第三步预览校验页，集中承载按帧对比、预览状态、输出决策和最终确认入队；确认任务不再是独立页面。
4. 直接访问 `/task-config` 或 `/preview` 时，如果当前没有源文件，会回到 `/source`，避免跳过第一步。
5. 新任务草稿默认使用 H.265 / libx265、CRF 码率模式和 medium preset；切换编码器后 preset 自动回退也优先选择 medium。
6. 真实 Tauri 预览中，源帧 proxy 和转码预览帧使用一致的预览缩放尺度，避免转码层因像素更多而显得比源片更清晰。
7. 单帧预览会禁用 2-pass 和 Dolby Vision 元数据保留；正式转码仍按当前参数执行完整链路。
8. Dolby Vision 元数据保留使用固定 runtime 中的 FFmpeg/FFprobe、x265 CLI 与 dovi_tool。首版支持单层 Profile 5 / compatibility 0 原链路保留，以及 Profile 8 / compatibility 1（Profile 8.1）HDR10 兼容链路；源片必须包含 BL+RPU、不能包含 EL，并使用 10-bit 恒定帧率视频。Profile 7 MEL/FEL、VFR、裁剪、缩放、变帧率和 2-pass 暂不进入该链路。
9. HDR10 和 HLG 源片在单帧预览中会优先映射为 SDR 图片，用于避免 WebView/PNG 预览因不支持 HDR 显示而出现偏色；预览映射会优先使用 `libplacebo`，缺少时再降级到 `zscale+tonemap`，并使用 ffprobe 读取到的色彩原色、传递函数、色彩矩阵和范围固定 `zscale` 输入端；Dolby Vision 预览 SDR 映射必须依赖 `libplacebo` 读取 RPU 元数据并输出 BT.709 SDR，不再把表面上兼容 BT.2020/PQ/BT.2020NC 的 DV 源片硬套到 `zscale+tonemap`；如果当前 FFmpeg 缺少对应 filter，或当前素材的颜色空间无法被可靠映射，预览会降级为未映射的普通预览帧，正式转码不受该预览映射影响。
10. 普通浏览器预览态提供只读示例数据和同源派生示例帧，仅用于设计 QA 与前端首屏验证；真实入队、样例写入和独立桌面窗口能力只在 Tauri 宿主内执行。
11. 开启 Dolby Vision 保留后，参数页会自动使用 H.265 / libx265 / CRF / 10-bit / MKV，并锁定完整时长、原分辨率和原帧率。后端按 profile 构建专用执行计划，转码结束后校验输出 DOVI profile、compatibility、BL/RPU/EL、分辨率、帧率、位深、RPU 帧数及逐帧动态元数据语义；校验通过后才原子发布最终文件。

### 4.2 参数方案库

参数方案库是用户长期复用的“编码方案资产”，不是简单模板列表。

职责：

1. 保存当前参数为方案。
2. 搜索、复制、删除方案。
3. 从方案进入工作台继续预览。
4. 记录最近使用时间和版本。
5. 未来支持按用途分类，例如“B 站上传”“归档压缩”“保留 HDR”。

当前代码映射：

1. `src/pages/TemplatesPage.tsx`
2. `src/components/workbench/TemplateDetailPanel.tsx`
3. `src-tauri/src/commands/templates.rs`
4. `src-tauri/src/models/template.rs`
5. `src-tauri/src/storage/repositories/templates_repo.rs`

调整建议：

1. 产品命名从“模板”调整为“参数方案”或“方案库”。
2. 模板保存的对象应继续只保存参数，不保存输入文件路径。
3. 方案库的主动作不是“管理”，而是“套用到工作台并验证”。

当前落地状态：

1. 方案库已按“方案资产库”组织首屏，强调用途、质量/速度取舍、最近使用和回到工作台验证。
2. 方案详情从字段罗列调整为“当前方案决策”，主动作改为“应用并回到工作台”。
3. 普通浏览器预览态会安全应用示例方案，不依赖 Tauri 命令；Tauri 宿主内仍调用真实 `apply_template`。

### 4.3 转码中心

转码中心承载队列、运行控制、历史记录和结果复盘。

职责：

1. 展示排队、运行、完成、失败和取消任务。
2. 按全局并发数执行 FIFO 调度。
3. 展示实时指标：进度、fps、speed、eta、pass 阶段。
4. 支持取消任务。
5. 展示实际命令、错误摘要、输入输出体积变化。
6. 删除历史记录，但不删除输出文件。

当前代码映射：

1. `src/pages/JobsPage.tsx`
2. `src/components/workbench/JobDetailPanel.tsx`
3. `src-tauri/src/transcode/job_manager.rs`
4. `src-tauri/src/commands/transcode.rs`
5. `src-tauri/src/models/job_history.rs`
6. `src-tauri/src/storage/repositories/jobs_history_repo.rs`

调整建议：

1. 产品命名从“任务中心”调整为“转码中心”。
2. 运行中任务和历史任务可以在同一模块，但需要明确两种视角：当前队列、历史复盘。
3. V1 先保持本机执行，不向用户暴露节点、传输和执行位置。

当前落地状态：

1. 转码中心首屏已从任务列表调整为“执行与复盘”，突出运行中、等待中、完成、失败四类状态。
2. 页面增加当前执行判断，明确用户下一步应关注队列空闲、运行指标还是失败排查。
3. 任务列表以状态、体积变化和结束时间组织，详情面板继续保留命令、错误、路径和控制能力。

### 4.4 环境与能力

环境与能力定义“这个应用当前能做什么”，影响工作台和转码中心的可用边界。

职责：

1. 探测 FFmpeg/FFprobe 是否可用。
2. 探测编码器能力和限制。
3. 管理全局并发数。
4. 管理默认输出目录。
5. 展示 Dolby Vision、AV1、VP9 等能力提示。

当前代码映射：

1. `src/pages/SettingsPage.tsx`
2. `src-tauri/src/commands/settings.rs`
3. `src-tauri/src/commands/probe.rs`
4. `src-tauri/src/probe/*`
5. `src-tauri/src/models/app_settings.rs`

调整建议：

1. 产品命名从“设置”调整为“环境与能力”或在设置页中强化该分区。
2. 能力探测结果应被工作台复用，避免页面间出现不同规则。
3. 并发数变化属于执行能力变化，不应被埋在普通偏好设置中。

当前落地状态：

1. 环境与设置首屏已调整为“环境能力”，把 FFmpeg/FFprobe、并发数和 Dolby Vision 保留能力解释成产品能力边界。
2. 运行环境和能力探测作为主内容展示，偏好设置和应用更新移到右侧辅助面板。
3. 普通浏览器预览态为更新检查提供安全预览反馈；Tauri 宿主内仍使用真实 updater。
4. Dolby Vision 能力探测同时展示 FFmpeg `dovi_rpu`、libx265 DV 参数、x265 CLI 与 dovi_tool；四项能力完整时才开放 P5/P8.1 RPU 保留。

### 4.5 节点与分布式

节点与分布式是 V2 模块，不进入 V1 主导航。

职责：

1. 管理 Local Node 和 Remote Node。
2. 管理远程文件入口。
3. 管理跨节点传输。
4. 管理分布式调度与失败迁移。

当前代码映射：

1. `docs/节点拓展与分布式转码设计.md`
2. `src/types/workbench.ts` 中的 `NodeDescriptor`、`TransferPlan`、`DistributedTask`
3. `src-tauri/src/models/node.rs`
4. `src-tauri/src/models/task.rs` 和 `src-tauri/src/models/job_history.rs` 中的 location 兼容字段

调整建议：

1. V1 只保留兼容字段，不新增用户入口。
2. 所有 `nodeId` 为空或缺失时继续解释为 `local`。
3. 等 V1 本机链路稳定后，再把“节点”作为独立模块加入导航。

## 5. 推荐导航结构

### 5.1 V1 导航

推荐 V1 只暴露四个一级入口：

1. `工作台`
2. `方案库`
3. `转码中心`
4. `环境与设置`

对应关系：

| 新模块 | 当前入口 | 核心用户意图 |
| --- | --- | --- |
| 工作台 | 源文件选择、任务配置、预览 | 选文件、调参、预览、确认任务 |
| 方案库 | 模板 | 复用和管理参数方案 |
| 转码中心 | 任务中心 | 查看队列、控制执行、复盘结果 |
| 环境与设置 | 设置 | 确认系统能力和全局运行参数 |

### 5.2 V2 导航

V2 在 V1 基础上增加：

1. `节点`
2. `文件入口`
3. `传输任务`

这些模块不应提前进入 V1 导航，否则会干扰当前本机转码主线。

## 6. 模块优先级

### 6.1 先收敛工作台

工作台决定用户能否完成第一条主链路。优先级最高。

应优先补齐：

1. 任务配置和预览之间的状态连续性。
2. 预览失败、FFmpeg 不可用、编码器不兼容时的解释。
3. 入队前的源文件、参数摘要和输出路径确认。
4. 从方案库套用后回到工作台的流程。

### 6.2 再增强转码中心

转码中心决定用户是否敢批量执行。

应优先补齐：

1. 队列筛选与状态分组。
2. 错误摘要和实际命令复制。
3. 输入输出体积变化展示。
4. 历史记录删除与输出文件关系说明。

### 6.3 最后整理方案库

方案库是复用能力，依赖工作台稳定。

应优先补齐：

1. 标签筛选。
2. 最近使用排序。
3. 方案详情中的参数摘要。
4. 直达预览入口。

## 7. 保持不变的约束

1. V1 仍然是本机桌面转码工具，不引入账号、云同步和远程节点入口。
2. 模板只保存参数，不保存具体输入文件路径。
3. 预览阶段可以降级 2-pass，但正式转码必须保持完整 2-pass 语义。
4. 转码队列仍使用全局并发数和 FIFO 调度。
5. 本地 JSON 仍是 V1 的持久化事实源。
6. 分布式字段必须保持向后兼容，旧数据缺少 location 时解释为本机。

## 8. 建议落地顺序

1. 文案层：把导航和页面标题调整为 `工作台 / 方案库 / 转码中心 / 环境与设置`。
2. 信息架构层：把 `任务配置` 与 `预览` 合并为工作台内步骤，而不是两个同级模块。
3. 业务对象层：在文档和代码命名中区分 `参数草稿`、`参数方案`、`转码任务`、`执行记录`。
4. 能力边界层：把 FFmpeg 探测、编码器能力和并发设置解释为“环境能力”。
5. V2 预留层：保留节点类型和 location 字段，但不在 V1 用户界面暴露。

## 9. 本次不建议做的事

1. 不建议直接重构后端模型，当前 `TaskConfig`、`Template`、`JobHistory` 已能支撑 V1。
2. 不建议把 V2 节点能力提前做成页面。
3. 不建议为了命名统一大规模移动文件。
4. 不建议把方案库做成复杂分类系统，V1 先补搜索、标签和最近使用即可。
5. 不建议把调参工作台拆得更细，用户需要的是连续调参体验。
