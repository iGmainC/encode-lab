# EncodeLab 节点拓展与分布式转码设计

- 文档版本：`v0.1`
- 更新时间：`2026-05-06`
- 文档类型：`AI 技术设计文档`
- 适用阶段：`V2 近期可实现设计`
- 关联规格：[`ffmpeg-v1-功能规格.md`](./ffmpeg-v1-功能规格.md)

## 1. 背景与目标

EncodeLab 当前以桌面端 UI、本机文件路径、本机 `ffmpeg` 进程和本地 JSON 队列为核心。节点拓展的目标是在不破坏现有单机体验的前提下，引入独立节点二进制，让 EncodeLab 可以管理远程文件入口和多节点转码任务。

第一版设计需要覆盖两类业务能力：

1. 远程入口模式：用户在 EncodeLab UI 中浏览某个节点上的文件，读取元数据、预览并发起转码；EncodeLab 主要作为 UI 和控制台。
2. 集群分发模式：EncodeLab 作为任务队列和调度中心，将多个文件的转码任务分配给多个节点，按需同步输入文件，并在完成后归集输出文件。

本设计只定义可落地的技术边界和演进方向，不表示 V1 已实现分布式调度。

## 2. 当前实现约束

当前实现中，以下能力都默认发生在 EncodeLab 主机本地：

1. 文件选择结果是本机绝对路径，前端和后端都以 `string` 路径传递输入、输出和预览文件。
2. 视频元数据读取通过本机 `ffprobe` 完成。
3. 预览帧生成和正式转码都通过本机 `ffmpeg` 子进程完成。
4. 任务队列由本机内存调度器管理，历史记录写入本机 `jobs-history.json`。
5. 输出路径由本机文件系统解析和自动避让重名。

节点化改造的第一步不是直接增加远程执行，而是先把“文件位置”和“执行位置”从本机路径中抽象出来，避免后续任务模型继续绑定单机语义。

## 3. 核心概念

### 3.1 Controller

Controller 是 EncodeLab 主控，负责 UI、节点管理、任务队列、调度决策、状态汇总和必要的流式中转。Controller 不应默认保存跨节点传输中的大文件。

### 3.2 Node Agent

Node Agent 是未来独立发布的节点二进制，运行在下游机器上。它负责暴露本机文件系统浏览、视频元数据探测、预览帧生成、转码执行、传输执行和状态上报能力。

### 3.3 Local Node

Local Node 是 EncodeLab 主机自身的内置节点。启用“主机参与计算”时，Controller 将本机作为一个特殊节点参与调度；Local Node 复用节点调度模型，但访问本机文件时不需要网络传输。

### 3.4 Remote Node

Remote Node 是通过 Node Agent 注册到 Controller 的远程节点。它必须定期心跳、上报能力和槽位，并接收 Controller 下发的文件、预览、转码和传输指令。

### 3.5 Location 与 Artifact

路径不再只表达为本机字符串，而应表达为“节点 + 节点内路径”：

```ts
/** 节点内文件位置。 */
type FileLocation = {
  /** 文件所在节点；local 表示 EncodeLab 主机内置节点。 */
  nodeId: string;
  /** 节点本机可访问的绝对路径。 */
  path: string;
};

/** 输入、输出或中间产物的位置描述。 */
type ArtifactLocation = FileLocation & {
  /** 文件用途。 */
  role: "input" | "output" | "preview" | "temp";
  /** 可选内容校验，用于跨节点复用和传输校验。 */
  checksum?: string;
  /** 文件大小，单位 byte。 */
  sizeBytes?: number;
};
```

### 3.6 Job、Task 与 Transfer

Job 是用户视角的一组转码工作，例如一季电视剧的 40 个文件。Task 是可调度的最小转码单元，例如单集视频的一次转码。Transfer 是文件移动或流式转发任务，独立于转码 Task 建模。

集群分发时，不能把“分配任务”简化为“远程执行 `ffmpeg`”。完整链路应是：

1. 判断输入文件在哪个节点。
2. 判断目标执行节点是否已有可复用输入。
3. 缺失时创建 Transfer。
4. Transfer 成功后创建或启动转码 Task。
5. 转码完成后按输出策略创建结果归集 Transfer。

## 4. 业务模式

### 4.1 远程入口模式

远程入口模式中，EncodeLab UI 面向用户提供类似远程文件浏览器的体验：

1. 用户选择一个节点作为文件来源。
2. UI 调用节点文件浏览接口展示目录、文件和基础属性。
3. 用户选中文件后，Controller 请求对应节点执行 `ffprobe`，并把元数据返回给 UI。
4. 预览阶段由文件所在节点生成预览帧，Controller 只接收帧路径引用或图片流。
5. 正式转码默认在文件所在节点执行；如果用户开启集群分发，才进入跨节点调度流程。

该模式下，EncodeLab 不需要把远程文件下载到主控才能开始操作。

### 4.2 集群分发模式

集群分发模式中，Controller 统一管理任务队列和调度策略。以输入文件都在 A 节点、B/C/D 为空闲算力节点为例：

1. Controller 将 Job 拆分为多个 Task。
2. Controller 按节点能力、空闲槽位和文件位置选择执行节点。
3. 若 Task 分配给 B/C/D，Controller 先创建从 A 到目标节点的 TransferPlan。
4. 输入文件到达目标节点后，目标节点执行转码。
5. 输出文件默认回收到源节点 A 的目标输出目录；用户也可以为 Job 指定其他输出位置。
6. 若用户启用主机参与计算，Local Node 作为额外节点参与调度。

## 5. 通信与注册

### 5.1 第一版通信模型

第一版默认采用 HTTP 命令接口 + WebSocket 事件通道：

1. HTTP 用于注册、文件浏览、元数据探测、任务下发、传输下发和控制命令。
2. WebSocket 用于心跳、任务进度、传输进度、节点状态和错误事件。
3. Controller 保存节点最近一次心跳、能力快照和运行槽位。

### 5.2 节点注册

注册流程采用 Controller 生成 token 的方式：

1. 用户在 EncodeLab 中创建节点接入 token。
2. 用户在节点机器启动 Node Agent，并传入 Controller 地址和 token。
3. Node Agent 调用注册接口，提交节点名称、平台、地址和能力摘要。
4. Controller 校验 token 后生成稳定 `nodeId`，并开始接收心跳。
5. token 可被吊销；节点离线后不应继续接收新任务。

第一版面向可信内网，token 是基础认证手段；暂不引入 mTLS、复杂用户权限或公网暴露模型。

## 6. 草案契约

以下类型用于约束未来实现方向，不要求立即落入 V1 代码。

```ts
/** 节点能力和运行状态。 */
type NodeDescriptor = {
  /** 节点唯一标识。 */
  id: string;
  /** 用户可读名称。 */
  name: string;
  /** local 表示主控内置节点，remote 表示独立 Agent。 */
  kind: "local" | "remote";
  /** 节点 HTTP 入口；local 节点可为空。 */
  endpoint?: string;
  /** 节点平台和架构。 */
  platform: "macos" | "linux" | "windows";
  /** 节点状态。 */
  status: "online" | "offline" | "draining" | "disabled";
  /** 节点可并发执行的转码槽位。 */
  slots: { total: number; used: number };
  /** ffmpeg、ffprobe、编码器和 GPU 能力摘要。 */
  capabilities: {
    ffmpegVersion?: string;
    encoders: string[];
    gpu?: string[];
  };
  /** 最近一次心跳时间。 */
  lastSeenAt?: string;
};

/** 跨节点文件传输计划。 */
type TransferPlan = {
  /** 传输任务 id。 */
  id: string;
  /** 源文件位置。 */
  source: FileLocation;
  /** 目标文件位置。 */
  target: FileLocation;
  /** 传输方式；relay 表示 Controller 零落盘流式中转。 */
  mode: "direct" | "relay";
  /** 当前状态。 */
  status: "queued" | "running" | "completed" | "failed" | "canceled";
  /** 已传输字节数。 */
  transferredBytes?: number;
  /** 失败原因。 */
  error?: string;
};

/** 可调度的单个转码单元。 */
type DistributedTask = {
  /** 任务 id。 */
  id: string;
  /** 所属用户级 Job。 */
  jobId: string;
  /** 输入文件位置。 */
  input: ArtifactLocation;
  /** 输出文件位置；未指定时默认使用输入源节点的输出目录。 */
  output: ArtifactLocation;
  /** 当前被分配的执行节点。 */
  assignedNodeId?: string;
  /** 转码参数快照。 */
  taskConfigSnapshot: unknown;
  /** 运行状态。 */
  status: "queued" | "preparing" | "transferring" | "running" | "completed" | "failed" | "canceled";
  /** 已尝试次数。 */
  attempt: number;
};

/** 节点上报事件。 */
type NodeEvent =
  | { type: "heartbeat"; nodeId: string; at: string; slots: { total: number; used: number } }
  | { type: "taskProgress"; nodeId: string; taskId: string; progress: number; fps?: number; speed?: number }
  | { type: "transferProgress"; nodeId: string; transferId: string; transferredBytes: number }
  | { type: "taskFailed"; nodeId: string; taskId: string; error: string }
  | { type: "nodeError"; nodeId: string; error: string };
```

## 7. 文件流转策略

### 7.1 节点直传优先

当源节点和目标节点可以互相访问时，Controller 只下发 TransferPlan，由节点之间直接传输文件。这样可以避免主控机器成为大文件传输瓶颈。

### 7.2 主控流式中转兜底

当节点之间无法互相访问时，Controller 可以进入 relay 模式：

1. 源节点向 Controller 打开文件读取流。
2. Controller 将数据流直接转发到目标节点写入接口。
3. Controller 不将文件写入本地临时目录，也不把中转数据作为持久事实源。
4. 中转过程仍上报 Transfer 进度和错误。

### 7.3 输出归集

输出位置由 Job 指定；若用户未指定，则默认回收到输入源节点的目标输出目录。对于“输入都在 A，任务分发到 B/C/D”的场景，B/C/D 完成后应将结果回传到 A。

## 8. 调度与失败策略

### 8.1 调度输入

Controller 做调度时至少参考以下信息：

1. Task 需要的编码器和参数能力。
2. 节点在线状态和可用槽位。
3. 输入文件当前所在节点。
4. 目标节点是否已有可复用输入文件。
5. 用户是否允许 Local Node 参与计算。
6. 传输成本和节点负载。

### 8.2 任务生命周期

分布式 Task 的推荐状态机：

```text
queued -> preparing -> transferring -> running -> completed
                         |             |
                         v             v
                       failed        failed
```

当执行节点已经拥有输入文件时，可以跳过 `transferring`。当任务失败但仍满足重试条件时，回到 `queued` 并增加 `attempt`。

### 8.3 失败重试与迁移

第一版默认采用有限重试后迁移：

1. 同一节点的临时失败可以有限重试。
2. 节点离线、传输失败或能力不再匹配时，任务回到队列并重新选择节点。
3. 如果所有节点都不满足能力要求，任务进入 `failed`，错误中记录缺失能力。
4. 单个 Task 失败不应阻塞同一 Job 中其他 Task 继续执行。

## 9. 状态机

### 9.1 节点状态

```text
online -> draining -> offline
online -> disabled
offline -> online
```

`draining` 表示不再接收新任务，但允许已有任务结束。`disabled` 表示用户主动禁用，不参与调度。

### 9.2 传输状态

```text
queued -> running -> completed
queued -> canceled
running -> failed | canceled
```

传输失败后，关联 Task 回到 `queued` 或进入 `failed`，由重试策略决定。

### 9.3 Job 状态

Job 是多个 Task 的聚合状态：

1. 所有 Task 完成时，Job 为 `completed`。
2. 部分 Task 失败且不可重试时，Job 为 `failed` 或 `partialFailed`。
3. 至少一个 Task 运行或传输时，Job 为 `running`。
4. 用户取消 Job 时，应取消未开始 Task、终止运行 Task，并中断关联 Transfer。

## 10. 演进路线

### 10.0 当前已落地的数据结构准备

当前代码已先行补充节点拓展所需的兼容字段和共享契约，但仍不启用远程执行：

1. Rust 后端新增 `src-tauri/src/models/node.rs`，集中定义 `FileLocation`、`ArtifactLocation`、`NodeDescriptor`、`TransferPlan`、`DistributedTask` 和 `NodeEvent`。
2. 前端 `src/types/workbench.ts` 同步补充节点、位置、传输和分布式任务类型。
3. `JobHistory` 保留旧字段 `inputFile` / `outputFile`，并新增可选 `inputLocation`、`outputLocation`、`executionNodeId` 和 `transferIds`；旧历史 JSON 缺少这些字段时仍按 `local + 原路径` 理解。
4. `OutputConfig` 新增可选 `location`，用于后续表达输出节点位置；为空时沿用当前本机输出目录语义。
5. 预览和命令构建请求新增可选 `inputLocation` / `outputLocation`，当前执行仍以字符串路径调用本机 `ffmpeg`，避免改变 V1 行为。

### 10.1 阶段一：抽象本机模型

1. 将输入、输出、预览文件从裸路径逐步迁移到 `FileLocation`。
2. 将本机执行器建模为 Local Node。
3. 让现有本机队列继续工作，不引入远程节点。

### 10.2 阶段二：远程入口

1. 增加节点注册和心跳。
2. 增加远程文件浏览、元数据探测和远程预览。
3. 支持在文件所在节点执行单文件转码。

### 10.3 阶段三：集群分发

1. 将 Job 拆分为可独立调度的 DistributedTask。
2. 增加 TransferPlan 和传输状态管理。
3. 增加节点直传和 Controller 流式中转。
4. 增加失败重试、任务迁移和输出归集。

## 11. 非目标

第一版不做以下能力：

1. 公网节点暴露和复杂权限系统。
2. mTLS、证书轮换和多租户隔离。
3. 共享存储强依赖；共享存储可作为未来优化，不作为第一版前提。
4. 节点二进制独立 UI。
5. 自动执行数据库迁移或重写现有 V1 任务存储。

## 12. 后续实现验收依据

后续代码改造应满足以下方向：

1. 本机路径可以逐步迁移为 `nodeId + path`。
2. 本机转码可以被建模为 `local node` 执行。
3. 远程文件选择不要求把文件下载到主控。
4. 多节点任务可以表达“源在 A，执行在 B/C/D，输出回 A”的完整链路。
5. 节点互通失败时，可以表达 Controller 零落盘流式中转兜底。
